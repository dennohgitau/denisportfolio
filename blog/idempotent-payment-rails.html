Designing Idempotent Payment Rails in Insurance: Pesapal, M-Pesa, and Bank Verification

Payment systems in the real world are noisy: users retry, callbacks arrive late, polling races happen, and providers occasionally return conflicting snapshots. In insurance, this can never create duplicate successful covers.

Common Failure Modes

  • Duplicate callbacks from providers after intermittent network failures.
  • User-triggered retries when redirect pages lag.
  • Asynchronous verification arriving after front-end timeout windows.

Idempotency Strategy

  • Database uniqueness rules around provider confirmation codes and successful final states.
  • Clear payment state machine transitions (pending, verifying, paid, failed, issued).
  • Server-side callback verification plus authoritative refresh via polling APIs.

Operational Guardrails

We maintained fast checkout while preserving audit quality by storing raw provider responses, normalizing key fields, and reconciling against internal state. This allows finance and underwriting to review evidence without guessing.
